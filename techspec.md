## 1. 기술 스택 ✅

| 영역 | 선택 | 이유 |
|---|---|---|
| 프레임워크 | Flutter | 크로스플랫폼(안드로이드+iOS 동시 출시), 기존 프로젝트('잠든 사이 숲')에서 다뤄본 경험 활용, 블록 캘린더 같은 커스텀 UI/드래그/중첩 구조 구현에 강점 |

## 2. 화면 on/off 감지 → "앱 포그라운드/백그라운드"로 재정의 ✅

**배경**: 안드로이드는 `BroadcastReceiver`로 화면 on/off를 직접 감지할 수 있지만, iOS는 앱이 백그라운드에 있을 때 시스템 차원의 화면 상태 감지를 허용하지 않음 (배터리/프라이버시 정책). 두 플랫폼에서 동일하게 동작하려면 다른 기준이 필요함.

**결정**: Flutter의 `AppLifecycleState`(resumed/paused)를 기준으로 타이머 자동 정지/재시작 트리거를 구현.

- 앱이 백그라운드로 전환(다른 앱 전환, 화면 잠금 등) → 자동 일시정지
- 앱이 포그라운드로 복귀 → **자동 일시정지였던 경우에만** 실행중으로 복귀 (수동 정지 상태는 포그라운드 복귀 여부와 무관하게 그대로 유지 — 기존 3단계 타이머 상태 로직 그대로 적용)

**장점**: 네이티브 코드 불필요, 두 플랫폼에서 동일하게 동작
**트레이드오프**: 안드로이드에서 "화면은 켜졌지만 앱은 아직 열지 않은 상태(잠금화면만 본 상태)"는 감지 불가 — 감수하기로 함

## 4. MVP 범위 제안 ⏸

네이티브 의존도를 기준으로 단계 구분 (확정 아님, 제안 단계):

- **1차**: 순수 Flutter로 구현 가능한 모든 기능 (계획표, 블록 중첩, 타이머, 알림, 잔디, 통계, 회고, 마감 작업 등)
- **2차(보류)**: 홈 화면 위젯, 잠금화면 타이머

## 5. 데이터 모델 설계 원칙 (개념 수준, 코드는 Claude Code에서) ✅

> ⚠️ **용어 구분**: "블록 템플릿"(아래 신규 항목, 재사용 가능한 활동 정의)과 "템플릿"(8번 항목 `반복 & 템플릿`, 하루/주 단위 일정 복제)은 이름이 비슷하지만 다른 개념. 코드/문서에서 혼동 주의.

- **블록은 "템플릿"과 "인스턴스"로 분리** (Phase 3 진행 중 발견되어 추가된 원칙) — `BlockTemplates`(제목·색상, 재사용 가능)와 `Blocks`(날짜·시작/종료시간·완료여부, `blockTemplateId` 참조)를 별도 엔티티로 관리. 사용자가 템플릿을 캘린더에 끌어다 놓을 때마다 새 인스턴스가 생성됨. 태그도 인스턴스가 아닌 템플릿에 부여(`BlockTags`가 `blockTemplateId` 참조)
- **타임블록과 체크리스트는 별도 엔티티로 분리** — 중첩 제약이 서로 다름(타임블록은 1단계만, 체크리스트는 무제한)이라 한 엔티티에 합치면 깊이 제한 검증이 까다로워짐
- **반복 일정은 "규칙"과 "실제 인스턴스"를 분리** — 규칙(반복 패턴) 자체와, 화면에 뜨는 각 날짜의 실제 블록(완료 체크·타이머 기록·잔디 표시가 인스턴스 단위로 필요)을 구분해서 관리
- **"목표 시간"은 별도 엔티티 없이 타임블록의 시작~종료 시간으로 계산** — 타임블록에 이미 있는 시작/종료 시간이 곧 계획 시간, 타이머 누적 시간이 실제 시간이므로 둘을 바로 비교
- **잔디(출석)도 별도 테이블 없이 계산** — 그날의 블록/체크리스트 완료 데이터를 기반으로 산출
- **마감 작업(5.11)은 타임블록과 완전히 별개의 엔티티** — 서로 다른 개념이므로 독립적으로 관리(이전 결정 사항 참고)
- **블록 체이닝(습관 스태킹용)** — `Blocks`에 `nextBlockId` 자기참조 필드로 연결 순서 표현 (Phase 1 구현 중 결정됨)
- 그 외 필요한 개념: 타이머 세션(한 블록에 여러 번 시작/정지 가능하므로 세션 단위 누적), 일정 템플릿(하루/주 단위 블록 구성 스냅샷)
